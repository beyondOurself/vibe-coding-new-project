# Vibe Coding 精华

> 摘自教程「20 项目实战」「30 经验技巧」：新项目怎么推进、哪些模版要常备、文末附一条完整走通的路径说明。

---

## 一、新项目从 0 到 1 的步骤

核心结论：**先规划，再写代码**。Vibe Coding 里 AI 是执行者，你不清「做什么」，它就会自由发挥，后期容易乱。

标准流程是 **5 步**：

1. **需求研究（Research）**  
   想清楚要解决什么问题、是否值得做、有没有同类产品可参考。把结论记进文档（如 `RESEARCH.md`），不必很正式。

2. **产品需求文档（PRD）**  
   把功能、优先级、界面期望、边界写清楚。PRD 是后续对话的「宪法」，AI 主要靠它理解意图。

3. **技术设计（Tech Design）**  
   写 `TECH_DESIGN.md`：技术栈、目录结构、数据模型、关键技术点。不熟悉选型可以直接问 AI。

4. **AI 代理指令（AGENTS.md）**  
   用社区常见的 `AGENTS.md`（也可用 `AI_RULES.md` 等），写清规范、风格、测试与注意事项，让各次生成保持一致。

5. **实现与迭代（Build）**  
   先搭能跑的框架，再按模块实现核心功能，最后打磨体验。每步小范围改、测、提交 Git。

**进度文档（与教程「上下文管理」一致）**：从第一天起维护 **`TODO.md`**（已完成 / 进行中 / 待开发 / 已知问题），功能做完或改需求就更新；另用 **`README.md`** 写清项目是什么、如何运行、目录与规范，新对话可贴给 AI 快速对齐。可与五步并行：**Build 阶段每完成一条就打勾**，避免 AI 和你重复劳动或漏做。

落地时常用 **三步迭代**：生成基础框架 → 逐个实现核心功能 → 优化细节与体验。复杂任务要拆小，避免一次让 AI 做完一整块业务。

**五步小例（记账应用）**

| 步骤 | 你具体在做什么 |
|------|----------------|
| Research | 发现市面 App 太复杂，你只要「记一笔 + 看列表 + 月汇总」 |
| PRD | 写清：收入/支出字段、列表排序、统计口径；图表、导出标成「后续」 |
| Tech Design | 定：React + Vite、LocalStorage、`Transaction` 含 amount/type/date |
| AGENTS.md | 写：只用函数组件、Tailwind、金额用 number 存 |
| Build | 先跑通「添加一条」→ 再列表 → 再统计，每步测完再下一步 |

## 二、建议固定下来的格式模版

下面这些在教程里反复出现，建议复制到自己的项目里按需改。

### 1）需求研究 `RESEARCH.md`（示例结构）

```markdown
# 项目名 - 需求研究

## 目标
（一句话：要解决什么问题）

## 调研发现
（市场/竞品/自己的约束）

## 核心需求
1. …
2. …
```

**填好的片段示例**

```markdown
# 番茄钟 - 需求研究

## 目标
工作 25 分钟休息 5 分钟，减少手机干扰。

## 调研发现
- 手机番茄钟广告多
- 我只需要网页版，上班开标签页即可

## 核心需求
1. 可改时长（默认 25/5）
2. 结束有提示音或浏览器通知
3. 数据仅存本机即可
```

### 2）用 AI 扩写 PRD 的提示（先写人话，再让模型结构化）

先写简短需求，例如：

```
我要做一个记账应用，能快速记录收支，查看每月统计。
要简单易用，不要复杂的功能。
```

再发给大模型：

```
请帮我把这个需求扩展成完整的产品需求文档（PRD），
要包含：
1. 产品概述和目标用户
2. 详细的功能列表
3. 功能优先级（MVP 和后续版本）
4. 界面设计要求
5. 技术栈建议
6. 非功能性需求（性能、安全等）
```

### 3）PRD 常见章节清单

- 产品概述  
- 目标用户  
- 核心功能（含 MVP / 后续）  
- 界面与交互期望  
- 技术栈与代码风格倾向  
- 限制条件与边界场景  

**PRD 里「MVP vs 后续」示例**

```markdown
### 必须做（MVP）
- 用户用邮箱+密码注册登录
- 发布一条带标题正文的帖子

### 后续版本
- Markdown、标签、全文搜索
```

### 4）技术设计 `TECH_DESIGN.md`（示例结构）

```markdown
# 项目名 - 技术设计

## 技术栈
（前端 / 后端 / 存储 / 部署）

## 项目结构
（目录树）

## 数据模型
（字段与类型）

## 关键技术点
（存储方式、鉴权、难点）
```

**数据模型示例**

```markdown
### Item（待办）
- id: string
- title: string
- done: boolean
- createdAt: string（ISO 日期）
```

### 5）`AGENTS.md`（示例结构）

```markdown
# 项目名 - AI 开发指令

## 项目概述

## 开发规范
（TS/框架/Hooks/样式方案等）

## 代码风格

## 测试要求

## 注意事项
（性能、安全、适配等）
```

若在 **Windows** 开发，可在 `AGENTS.md` 或对话里写明：操作系统、终端（如 PowerShell）、要求使用 Windows 兼容命令。

```markdown
## 开发环境
- 操作系统：Windows 11
- 终端：PowerShell
- 包管理：npm；请勿使用仅适用于 bash 的一键脚本，改用跨平台写法或分步命令
```

### 6）`TODO.md`（任务清单）与 `README.md`

教程里强调：**文档体系 = README + TODO + 上下文文件（如 AGENTS）**。`TODO.md` 让「人和 AI」都看到进度，减少失忆与重复。

**应包含四块**

1. 已完成  
2. 进行中（可拆子任务）  
3. 待开发  
4. 已知问题（Bug / 待优化）

**示例骨架**

```markdown
# 开发进度

## 已完成 ✅
- [x] 项目初始化
- [x] …

## 进行中 🚧
- [ ] 当前功能名
  - [x] 子步骤
  - [ ] …

## 待开发 📋
- [ ] …

## 已知问题 🐛
- …
```

**让 AI 更新 TODO 的提示**

```markdown
我们刚完成了【功能名】。请更新 TODO.md：把该项标为已完成，并调整「进行中」与「待开发」。
```

### 7）初始化项目的对话模版

```
请根据 PRD.md、TECH_DESIGN.md 和 AGENTS.md 的要求，
初始化项目并创建基本的项目结构，包括：
1. 安装必要的依赖
2. 创建目录结构
3. 配置开发环境
4. 创建基础的路由和页面框架
确保项目能够正常启动。
```

### 8）实现单个功能的对话模版

```
我要实现【功能名】，请根据 PRD.md 和 TECH_DESIGN.md 中的要求，
【具体说明文件/组件名与行为】。
```

**示例**

```
我要实现「添加待办」功能。请根据 PRD.md 和 TECH_DESIGN.md：
在 src/components/TodoInput.tsx 中增加输入框与提交按钮，
提交时在父组件中往 todos 列表追加一条 Item，并清空输入框。
```

### 9）限定修改范围（避免改 A 坏 B）

```
仅修改 path/to/File.tsx：
1. …
2. 保持现有的样式和布局
3. 不要改动其他文件
```

**示例**

```
仅修改 src/components/AddTransaction.tsx：
1. 在提交前校验金额必须大于 0
2. 保持现有 Tailwind 类名与布局不变
3. 不要改动其他文件
```

### 10）需求描述框架（基础版五要素）

1. **是什么**：要做哪个功能或组件  
2. **为什么**：目的  
3. **怎么做**：技术或实现约束  
4. **什么样**：视觉与交互  
5. **何时何地**：使用场景与位置  

进阶可补第 6 点：**受众**（给谁用、技术水平）。

**同一需求，五要素串起来一例**

- **是什么**：文章列表页的搜索框  
- **为什么**：文章多的时候快速定位  
- **怎么做**：Next.js App Router，输入防抖 300ms，调 `/api/posts?q=`  
- **什么样**：圆角输入框、左侧放大镜图标、占位符「搜索文章…」  
- **何时何地**：桌面与移动端顶栏右侧；无结果时显示「暂无匹配」  
- **受众（可选）**：普通读者，不需要高级筛选  

### 11）对话工程里的四类快捷模版

**新功能**

```markdown
我要开发一个新功能：【功能描述】。我的技术栈是【技术栈】。请帮我：
1）分析这个功能的核心需求
2）建议一个合理的实现方案
3）列出可能遇到的问题
```

**调试**

```markdown
我遇到了一个问题：【问题描述】。报错信息是：【错误信息】。相关代码是：【代码片段】。请帮我：
1）分析可能的原因
2）给出解决方案
3）解释为什么会出现这个问题
```

**优化代码**

```markdown
这是我的代码：【代码片段】。它的功能是【功能说明】。请帮我：
1）找出可能的性能问题
2）改进代码的可读性
3）指出潜在的 bug
```

**学习概念**

```markdown
我想学习【技术/概念】。请：
1）用简单的语言解释它是什么
2）给我一个实际的使用例子
3）告诉我什么时候应该用它
```

**让 AI 先问你再答**

```markdown
我想做【你的需求】。请你在回答前，先问我几个问题，了解更多细节，然后再给出方案。
```

**四类模版各填一版（可直接改括号）**

```markdown
我要开发一个新功能：个人主页展示最近 5 条 GitHub 动态。技术栈是 Next.js 14 + TS。请帮我：
1）分析核心需求
2）建议实现方案（是否用 Octokit、是否 SSR）
3）列出可能的问题（API 限流、Token 放哪）
```

```markdown
我遇到了一个问题：本地登录成功，部署到 Vercel 后 Cookie 丢失。
报错信息：【粘贴控制台或 Network】。
相关代码：middleware.ts 里【粘贴片段】。
请分析原因、给出改法、解释原理。
```

```markdown
这是我的代码：【useMemo 依赖里放了整个 user 对象】。功能是根据 user.id 拉取详情。
请：1）是否多余渲染 2）如何改依赖与可读性 3）若 user 为 null 有无隐患。
```

```markdown
我想学习 React 的 useCallback。请：1）一句话说明它和 useMemo 区别 2）给一个「子组件用 memo 时父组件传函数」的例子 3）什么时候不必用 useCallback
```

### 12）好 Prompt 与差 Prompt 对照（原则）

| 差 | 好 |
|----|-----|
| 帮我做一个按钮 | 用 React + Tailwind 写一个 `<Button>`：三种 variant（primary/secondary/danger）、支持 `loading` 时显示转圈并禁用点击、`disabled` 时降低透明度、点击时有轻微 scale 动效 |
| 实现用户管理 | 第一步：只定义 `User` 类型与内存里的假数据列表；第二步再做列表页只读展示。请先完成第一步 |
| 加一个用户友好的错误提示 | 当邮箱正则不通过时，在输入框下方用 `text-red-500 text-sm` 显示「请输入有效的邮箱地址」，且不影响已通过的字段 |

## 三、与步骤配套的心法（便于记忆）

- **Planning is Everything**：规划比堆代码更重要。  
  *例：先写 PRD 里「只做列表+详情」，再开发；比边聊边加「评论、点赞、私信」更不易返工。*  
- **MVP**：先最小可用，再扩展。  
  *例：记账 MVP 只要「记一笔、看列表、看本月合计」；导出 Excel 放到 v2。*  
- **小步迭代**：完成一步、测试一步，再下一步。  
  *例：登录页先做静态表单 → 再加校验 → 再对接接口，每步能点能测。*  
- **上下文**：新会话补技术栈与 PRD 要点；大项目靠模块化减上下文负担。  
  *例：新开对话先贴一句「本项目 Vue3 + Pinia + Vue Router，见 PRD 第三节」。*  
- **产品化表达**：需求具体、可验收，少抽象形容词。  
  *例：把「好看」改成「卡片圆角 12px、阴影 sm、主色 #2563EB」。*

## 四、完整案例说明：个人作品集网站（从想法到可开发）

下面用教程里的「个人作品集」思路，串起 **5 步 + 文档模版**，便于你对照自己做其他项目。

**背景**：需要一个能展示个人介绍、技能、项目与联系方式的站点，风格现代、深色、响应式，最终可部署到 Vercel。

1. **Research**  
   浏览他人 portfolio，记下喜欢的布局与动效，写入 `RESEARCH.md`（目标用户是自己与招聘方，核心是「清晰展示 + 好看」）。

2. **PRD（节选结构）**  
   - 首页：大标题、简介、头像  
   - 关于我：详细介绍、技能列表  
   - 项目：卡片（名称、截图、描述、技术栈、链接）  
   - 联系方式：邮箱、GitHub、社交链接  
   - 设计：简洁现代、深色主题、滚动动效、移动端适配  

3. **Tech Design（节选）**  
   - 技术栈：React + TypeScript + Vite、Tailwind CSS、可选 React Router、Framer Motion  
   - 结构：`components` 下放 Header/Hero/About/Projects/Contact/Footer，`data` 下放 `projects.ts`、`skills.ts`  
   - 数据：项目与技能用 TS 文件中的数组维护  

4. **AGENTS.md（节选）**  
   - 函数式组件 + Hooks、Tailwind 写样式、组件可复用  
   - 深色主题示例色值、强调色渐变、平滑滚动、移动端适配  
   - 图片懒加载、链接可点、避免过度设计  

5. **Build**  
   - 先让 AI 按三份文档初始化工程并跑通  
   - 再分块实现各区块与数据接入  
   - 最后调动画、适配与性能  

**对话可怎么拆（与 Build 对应）**

1. 「按 PRD + TECH_DESIGN + AGENTS 初始化 Vite+React 项目，先只渲染 Hero 占位」→ 本地 `npm run dev` 能打开  
2. 「实现 About 区块，数据从 `skills.ts` 读」→ 浏览器里段落与列表正确  
3. 「Projects 区用 `projects` 数组 map 出卡片，图片用懒加载」→ 检查移动端不换行炸版  

按同一套路，把「作品集」换成「记账」「待办」等，只需替换 Research/PRD 里的业务内容，**模版文件名与 5 步顺序可以不变**，这就是教程强调的、可复用的 Vibe Coding 工作方式。
